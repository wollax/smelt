<objective>

Purpose: Users need to see the merge order before committing to execution. The `merge plan` subcommand is the primary UX for merge order intelligence — it shows file sets, overlap scores, and the computed order. This is the user-facing surface for the algorithms built in Plan 02.
Output: Updated CLI with merge subcommands, comfy-table plan output, JSON output, integration tests.
</objective>

       Update the match arm in `run()`:
       ```rust
       Some(Commands::Merge { command }) => {
           let git = GitCli::new(git_binary, repo_root.clone());
           match command {
               commands::merge::MergeCommands::Run {
                   manifest,
                   target,
                   strategy,
               } => {
                   commands::merge::execute_merge_run(git, repo_root, &manifest, target, strategy)
                       .await
               }
               commands::merge::MergeCommands::Plan {
                   manifest,
                   target,
                   strategy,
                   json,
               } => {
                   commands::merge::execute_merge_plan(git, repo_root, &manifest, target, strategy, json)
                       .await
               }
           }
       }
       ```

    2. In `crates/smelt-cli/src/commands/merge.rs`, restructure the module:

          use clap::Subcommand;

       b. Define the `MergeCommands` enum:
          ```rust
          #[derive(Subcommand)]
          pub enum MergeCommands {
              /// Execute the merge pipeline
              Run {
                  /// Path to the session manifest file
                  manifest: String,
                  /// Override target branch name
                  #[arg(long)]
                  target: Option<String>,
                  /// Merge ordering strategy (completion-time or file-overlap)
                  #[arg(long, value_parser = parse_strategy)]
                  strategy: Option<MergeOrderStrategy>,
              },
              /// Preview the merge order without executing
              Plan {
                  /// Path to the session manifest file
                  manifest: String,
                  /// Override target branch name (used for display)
                  #[arg(long)]
                  target: Option<String>,
                  /// Merge ordering strategy (completion-time or file-overlap)
                  #[arg(long, value_parser = parse_strategy)]
                  strategy: Option<MergeOrderStrategy>,
                  /// Output as JSON instead of table
                  #[arg(long)]
                  json: bool,
              },
          }
          ```

       c. Add a parser function for `MergeOrderStrategy` from CLI string:
          ```rust
          fn parse_strategy(s: &str) -> Result<MergeOrderStrategy, String> {
              match s {
                  "completion-time" => Ok(MergeOrderStrategy::CompletionTime),
                  "file-overlap" => Ok(MergeOrderStrategy::FileOverlap),
                  _ => Err(format!("unknown strategy '{s}' (expected: completion-time, file-overlap)")),
              }
          }
          ```

       d. Rename existing `execute_merge` to `execute_merge_run` and update its signature:
          ```rust
          pub async fn execute_merge_run(
              git: GitCli,
              repo_root: PathBuf,
              manifest_path: &str,
              target: Option<String>,
              strategy: Option<MergeOrderStrategy>,
          ) -> anyhow::Result<i32>
          ```
          - Build MergeOpts with both target and strategy:
            ```rust
            let opts = MergeOpts {
                target_branch: target,
                strategy,
            };
            ```
          - Keep all existing error handling.
          - After successful merge, if `report.plan` is Some, print the strategy used:
            ```rust
            if let Some(ref plan) = report.plan {
                eprintln!("Strategy: {:?}{}", plan.strategy,
                    if plan.fell_back { " (fell back to completion-time)" } else { "" });
            }
            ```

       e. Implement `execute_merge_plan`:
          ```rust
          pub async fn execute_merge_plan(
              git: GitCli,
              repo_root: PathBuf,
              manifest_path: &str,
              target: Option<String>,
              strategy: Option<MergeOrderStrategy>,
              json: bool,
          ) -> anyhow::Result<i32>
          ```
          - Load manifest.
          - Create MergeRunner.
          - Call a new method `MergeRunner::plan()` that runs collect_sessions + order_sessions but NOT merge_sessions. This method returns `Result<MergePlan>`.
            **Important:** You will need to add a `pub async fn plan(&self, manifest: &Manifest, opts: MergeOpts) -> Result<MergePlan>` method to MergeRunner in `crates/smelt-core/src/merge/mod.rs`. This method:
            1. Validates .smelt/ exists
            2. Calls collect_sessions()
            3. Resolves strategy
            4. Calls order_sessions()
            5. Returns the MergePlan
            (This does NOT create branches, worktrees, or perform merges.)
          - If `json` is true, serialize MergePlan to JSON with serde_json::to_string_pretty and print to stdout.
          - If `json` is false, call `format_plan_table(&plan)` and print to stdout.

       f. Implement `format_plan_table(plan: &MergePlan) -> String`:
          - Use comfy-table with UTF8_FULL preset and UTF8_ROUND_CORNERS modifier.
          - Section 1: "Merge Order" table with columns: #, Session, Files Changed, Strategy Note
          - Section 2: If file-overlap strategy, "Pairwise File Overlap" table with columns: Session A, Session B, Overlapping Files, Count
          - Section 3: Per-session file list (collapsible — show first 10 files, then "... and N more")
          - If fell_back, add a note line: "Note: File overlap strategy could not differentiate sessions — fell back to manifest order."

    3. Error handling in `execute_merge_plan`: match on SmeltError variants the same way execute_merge_run does, but only the subset that can occur during planning (NotInitialized, SessionError, NoCompletedSessions, ManifestParse).
