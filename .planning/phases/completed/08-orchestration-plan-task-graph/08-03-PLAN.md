**1. Create `crates/smelt-cli/src/commands/orchestrate.rs`** with:

- `OrchestrateCommands` enum (clap Subcommand):
  ```rust
  #[derive(Subcommand)]
  pub enum OrchestrateCommands {
      /// Execute an orchestration plan
      Run {
          /// Path to the manifest TOML file
          manifest: String,
          /// Override merge target branch name
          #[arg(long)]
          target: Option<String>,
          /// Merge ordering strategy (completion-time, file-overlap)
          #[arg(long)]
          strategy: Option<MergeOrderStrategy>,
          /// Show verbose output including session logs
          #[arg(long)]
          verbose: bool,
          /// Disable AI conflict resolution
          #[arg(long)]
          no_ai: bool,
          /// Output results as JSON
          #[arg(long)]
          json: bool,
      },
  }
  ```

- `execute_orchestrate_run()` function:
  1. Load manifest from file path. Also read raw content for hash computation.
  2. Set up CancellationToken and Ctrl-C signal handler:
     ```rust
     let cancel = CancellationToken::new();
     let cancel_clone = cancel.clone();
     tokio::spawn(async move {
         tokio::signal::ctrl_c().await.ok();
         eprintln!("\nInterrupted — shutting down gracefully...");
         cancel_clone.cancel();
     });
     ```
  3. Check for resume: Use `RunStateManager::find_incomplete_run()`. If found:
     - Verify manifest hash matches. If not, warn that manifest changed since last run.
     - If stderr is a TTY, prompt with `dialoguer::Confirm`: "Previous incomplete run found (<run-id>). Resume? [Y/n]"
     - If not a TTY, skip resume (start fresh)
     - If user says yes, call `orchestrator.resume()` instead of `run()`
  4. Build conflict handler: Reuse the same `build_conflict_handler()` pattern from `commands/merge.rs` — check no_ai, TTY, AiConfig, build MergeConflictHandler enum variant.
  5. Set up dashboard:
     - If stderr is a TTY and not --json:
       - Create `indicatif::MultiProgress` on stderr
       - Create one `ProgressBar::new_spinner()` per session
       - Styles:
         - Pending: `"  {prefix:.dim} waiting"`
         - Running: `"  {spinner:.green} {prefix} running ({elapsed})"`  with `tick_strings` for animation
         - Completed: `"  {prefix:.green} done ({elapsed})"`
         - Failed: `"  {prefix:.red} failed"`
         - Skipped: `"  {prefix:.yellow} skipped"`
         - Cancelled: `"  {prefix:.dim} cancelled"`
       - Store `HashMap<String, ProgressBar>` for the `on_status` callback
     - If not a TTY or --json:
       - Use simple `eprintln!` line-by-line fallback: `"[session-name] status"` format
  6. Create `on_status` callback closure that updates the dashboard progress bars based on `SessionRunState` transitions.
  7. Create `Orchestrator`, call `run()` or `resume()`.
  8. On success:
     - If --json: serialize `OrchestrationReport` to stdout via `serde_json::to_string_pretty`
     - Otherwise: render summary table with comfy-table:
       - Header: "Orchestration Report: <manifest-name>"
       - Session results table: Name | Status | Duration
       - Merge summary (if merge happened): reuse merge report display from commands/merge.rs
       - Footer: "Completed in X.Xs"
  9. On error: print error to stderr, return exit code 1.
  10. Return exit code 0 on success.

**1. Create `crates/smelt-cli/tests/cli_orchestrate.rs`** with integration tests:

Use the same test infrastructure pattern as `cli_merge.rs` and `cli_session.rs`:
- `setup_test_repo()` helper that creates a git repo as a subdirectory of tempdir (for worktree cleanup)
- `write_manifest()` helper that writes TOML to a file
- Tests use `assert_cmd::Command` to run the smelt binary

**Test cases:**

- `orchestrate_two_parallel_sessions`:
  - Manifest with 2 independent scripted sessions (parallel_by_default=true, no depends_on)
  - Both sessions create different files (no conflicts)
  - Verify: exit code 0, output contains both session names, merged branch exists with files from both sessions

- `orchestrate_sequential_dependency`:
  - Manifest with A->B dependency (B depends_on A)
  - Both create non-overlapping files
  - Verify: exit code 0, both sessions completed, merge successful

- `orchestrate_skip_dependents_on_failure`:
  - Manifest with A->B, A has simulate_failure="crash"
  - on_failure = "skip-dependents"
  - Verify: exit code 1 (partial failure), output shows A failed and B skipped

- `orchestrate_abort_on_failure`:
  - Manifest with A (simulate_failure="crash"), B independent
  - on_failure = "abort"
  - Verify: exit code 1, output shows abort message

- `orchestrate_json_output`:
  - Same as two_parallel_sessions but with --json flag
  - Verify: stdout is valid JSON, contains session results and merge report

- `orchestrate_diamond_dependency`:
  - A -> {B, C} -> D pattern
  - All sessions create non-overlapping files
  - Verify: exit code 0, all 4 sessions completed, merge has all files

- `orchestrate_implicit_sequential`:
  - parallel_by_default = false, no depends_on
  - 3 sessions run in manifest order
  - Verify: exit code 0, all completed

**All tests:**
- Use `NO_COLOR=1` or pass `--no-color` to avoid ANSI in test output
- Run with `assert_cmd` not `cargo run`
- Verify exit codes explicitly
- Check stderr for status messages OR stdout for JSON output
