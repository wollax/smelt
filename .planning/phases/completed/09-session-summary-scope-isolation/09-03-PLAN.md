**1. Create `crates/smelt-cli/src/commands/summary.rs`** with:

- `SummaryCommands` — this is NOT a subcommand enum since `smelt summary` has no subcommands. Instead, use a struct or handle directly:
  ```rust
  /// CLI arguments for the summary command.
  /// Usage: `smelt summary <manifest.toml> [--run-id <id>] [--json] [--verbose]`
  #[derive(clap::Args)]
  pub struct SummaryArgs {
      /// Path to the manifest file (used to derive manifest name for run lookup)
      pub manifest: String,
      /// Show summary for a specific run ID (defaults to latest completed run)
      #[arg(long)]
      pub run_id: Option<String>,
      /// Output as JSON
      #[arg(long)]
      pub json: bool,
      /// Show verbose per-session file lists
      #[arg(long)]
      pub verbose: bool,
  }
  ```

- `execute_summary()` function:
  1. Determine the run_id:
     - Load the manifest file from the required `manifest` positional argument to derive the manifest name.
     - If `--run-id` provided, use it directly.
     - Otherwise, call `RunStateManager::find_latest_completed_run(manifest_name)` to find the most recent completed run.
     - If no completed run found, print error and return exit code 1.
  2. Load summary via `RunStateManager::load_summary(run_id)`.
  3. If `--json`: serialize `SummaryReport` to stdout via `serde_json::to_string_pretty`, return 0.
  4. Otherwise: call `format_summary_table()` for human-readable output.
  5. If `--verbose`: call `format_summary_verbose()` for detailed per-session file lists.
  6. Print to stdout.

- `format_summary_table(report: &SummaryReport) -> String`:
  - Compact comfy-table:
    ```
    Session Summary: <manifest_name>

    ╭─────────────────────┬───────┬────────┬────────╮
    │ Session             │ Files │ +Lines │ -Lines │
    ├─────────────────────┼───────┼────────┼────────┤
    │ database-migration  │ 3     │ 120    │ 5      │
    │ backend-api         │ 8     │ 340    │ 22     │
    │ frontend-ui         │ 5     │ 210    │ 8      │
    ├─────────────────────┼───────┼────────┼────────┤
    │ Total               │ 16    │ 670    │ 35     │
    ╰─────────────────────┴───────┴────────┴────────╯
    ```
  - Use comfy-table `UTF8_FULL` + `UTF8_ROUND_CORNERS` preset (matching existing convention).
  - Add a totals row at the bottom.

- `format_violations(report: &SummaryReport) -> Option<String>`:
  - Returns `None` if no violations (section omitted entirely).
  - Otherwise:
    ```
    Scope Violations:
      session-a modified 2 file(s) outside its declared scope:
        - src/db/schema.rs (scope: src/auth/**)
        - README.md (scope: src/auth/**)
      session-b modified 1 file(s) outside its declared scope:
        - Cargo.lock (scope: src/api/**)
    ```
  - Tone is neutral/factual per CONTEXT.md decisions.

- `format_summary_verbose(report: &SummaryReport) -> String`:
  - Per-session blocks showing all files with line counts:
    ```
    Session: database-migration (3 files, +120, -5)
      src/db/schema.rs          +80  -2
      src/db/migrations/001.sql +35  -0
      src/db/mod.rs             +5   -3

    Session: backend-api (8 files, +340, -22)
      ...
    ```
  - Commit messages listed after file list:
    ```
      Commits:
        - Add database schema
        - Add migration 001
    ```

  ```rust
      }
  }
  ```

**1. Create `crates/smelt-cli/tests/cli_summary.rs`** with integration tests:

Use the same test infrastructure pattern as `cli_orchestrate.rs`:
- `setup_test_repo()` helper that creates a git repo as a subdirectory of tempdir
- `write_manifest()` helper that writes TOML to a file
- Tests use `assert_cmd::Command` to run the smelt binary

**Test cases:**

- `orchestrate_shows_summary_table`:
  - Manifest with 2 sessions creating different files, each with file_scope set
  - Run `smelt orchestrate run <manifest>`
  - Verify: exit code 0, stderr/stdout contains "Session Summary" or similar, contains both session names with file counts

- `orchestrate_shows_scope_violations`:
  - Manifest with 1 session:
    - file_scope = ["src/auth/**"]
    - Script creates files in both src/auth/ AND src/db/ (out of scope)
  - Run `smelt orchestrate run <manifest>`
  - Verify: exit code 0 (violations are advisory), output contains "Scope Violations", contains the out-of-scope file path

- `orchestrate_no_violations_section_when_clean`:
  - Manifest with 1 session:
    - file_scope = ["src/**"]
    - Script only creates files under src/
  - Run `smelt orchestrate run <manifest>`
  - Verify: exit code 0, output does NOT contain "Scope Violations"

- `orchestrate_no_violations_when_no_file_scope`:
  - Manifest with 1 session, no file_scope set at all
  - Script creates files anywhere
  - Verify: output does NOT contain "Scope Violations" (opt-in behavior)

- `orchestrate_shared_files_not_flagged`:
  - Manifest with shared_files = ["Cargo.toml", "Cargo.lock"]
  - Session with file_scope = ["src/auth/**"]
  - Script modifies src/auth/login.rs AND Cargo.toml
  - Verify: no violations (Cargo.toml is in shared_files)

- `orchestrate_json_includes_summary`:
  - Same as basic test but with --json flag
  - Verify: stdout is valid JSON, contains "summary" key with session summaries and totals

- `standalone_summary_command`:
  - First run `smelt orchestrate run <manifest>` to produce a completed run
  - Then run `smelt summary <manifest.toml>` — manifest is a required positional argument
  - Verify: exit code 0, output contains summary table
  - Note: summary.json persists in the run directory after orchestration completes (see Plan 02 persistence invariant)

**All tests:**
- Use `NO_COLOR=1` to avoid ANSI in test output
- Run with `assert_cmd` not `cargo run`
- Verify exit codes explicitly
- Manifests use scripted sessions for deterministic file changes
