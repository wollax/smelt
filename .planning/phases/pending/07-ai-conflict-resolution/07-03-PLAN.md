**3. Create `AiInteractiveConflictHandler`** struct in `merge.rs`:

This is a composite handler that attempts AI resolution first, shows the result to the user, and falls back to the existing `InteractiveConflictHandler` if AI fails or is rejected.

```rust
struct AiInteractiveConflictHandler<G: GitOps, P: AiProvider> {
    ai_handler: AiConflictHandler<G, P>,
    provider: P,     // also held directly for retry calls that bypass ai_handler
    config: AiConfig,
    verbose: bool,
}
```
Note: `provider` must be `Clone` (or wrapped in `Arc`) since `AiConflictHandler` also holds a reference. Use `Arc<P>` if `P` is not `Clone`, or require `P: Clone`. The simplest approach: have both `AiConflictHandler` and `AiInteractiveConflictHandler` hold `Arc<P>` â€” update `AiConflictHandler::new` in Plan 02 accordingly to accept `Arc<P>` or `P: Clone`.

`impl ConflictHandler for AiInteractiveConflictHandler`:
- `handle_conflict()`:
  1. **Save "ours" content** per file before AI resolution: read `work_dir.join(file)` for each file and store as `Vec<(String, String)>` (file_path, original_content). Actually, better: extract the "ours" version via `git show :2:file` â€” but the handler doesn't have direct git access. Instead, save the current conflicted file content before AI modifies it (this is the working tree with markers). OR: since the AI handler writes resolved content, read the file AFTER AI resolution to get the AI output, and compare against the ours version. The simplest approach: before calling AI handler, read the conflicted file content (with markers). After AI handler succeeds, read the resolved file content. Show diff of `ours` (from index stage 2 via the AI handler's git) vs `resolved`.

  Actually, the cleanest approach since `AiInteractiveConflictHandler` has access to the `ai_handler` which has `git`:
  - Before AI: for each file, read the current working tree content (contains markers) and store it.
  - Call `self.ai_handler.handle_conflict(...)`.
  - If Err: print the error, fall through to manual handler.
  - If Ok(Resolved): for each file, read the new content from disk. Show a colored unified diff of the marker-containing original vs the AI-resolved version using `similar::TextDiff`. Then prompt Accept/Edit/Reject.

  2. **Diff display** per file using `similar`:
  ```rust
  fn format_colored_diff(original: &str, resolved: &str, filename: &str) -> String {
      use similar::TextDiff;
      let diff = TextDiff::from_lines(original, resolved);
      let unified = diff.unified_diff()
          .context_radius(3)
          .header(&format!("a/{filename} (conflicted)"), &format!("b/{filename} (ai-resolved)"))
          .to_string();
      // Color the output using console
      let mut colored = String::new();
      for line in unified.lines() {
          if line.starts_with('+') && !line.starts_with("+++") {
              colored.push_str(&format!("{}\n", console::style(line).green()));
          } else if line.starts_with('-') && !line.starts_with("---") {
              colored.push_str(&format!("{}\n", console::style(line).red()));
          } else if line.starts_with("@@") {
              colored.push_str(&format!("{}\n", console::style(line).cyan()));
          } else {
              colored.push_str(&format!("{line}\n"));
          }
      }
      colored
  }
  ```

  3. **Accept/Edit/Reject prompt** using dialoguer::Select (same pattern as existing InteractiveConflictHandler):
  ```
  AI proposed resolution for N file(s):
    [diff output per file]

  > Accept (use AI resolution as-is)
  > Edit (open files for manual tweaking, then continue)
  > Reject (provide feedback for retry, or fall back to manual)
  ```

  4. **Accept**: Return `ConflictAction::Resolved(ResolutionMethod::AiAssisted)`. The merge loop will re-scan for markers â€” if the AI output is clean, it proceeds. The `ConflictAction::Resolved(ResolutionMethod)` change was already made in Plan 02, so use it directly here.

  5. **Edit**: Tell the user "Edit the resolved files in {work_dir}, then press Enter to continue." Wait for Enter (dialoguer::Confirm or just stdin readline). Then return `ConflictAction::Resolved(ResolutionMethod::AiEdited)`.

  6. **Reject**: Prompt for optional feedback text (dialoguer::Input with allow_empty). Increment retry counter. If retries < max_retries:
     - **Retry mechanism:** `AiInteractiveConflictHandler` calls `build_resolution_prompt` and `build_retry_prompt` directly (imported from `smelt_core::ai`), then calls `provider.complete()` directly â€” bypassing `AiConflictHandler::handle_conflict()`. This is necessary because the retry prompt must incorporate user feedback, and `AiConflictHandler` treats each call as a fresh single-attempt resolve. The `AiInteractiveConflictHandler` holds the same `provider` and `config` references, so it can:
       1. For each file, call `build_resolution_prompt(...)` to get the original prompt
       2. Call `build_retry_prompt(&original_prompt, &feedback)` to append feedback
       3. Call `provider.complete(model, build_system_prompt(), &retry_prompt).await`
       4. Write resolved content to disk via `tokio::fs::write`
     - Show diff again, re-prompt Accept/Edit/Reject
     If retries exhausted:
     - Print "AI retries exhausted, falling back to manual resolution..."
     - Restore the original conflicted files (write back the saved marker content)
     - Delegate to `InteractiveConflictHandler::handle_conflict()` (but return Manual method)
     - Return whatever the interactive handler returns, but change Resolved method to Manual

  7. **API failure handling**: If `self.ai_handler.handle_conflict()` returns `Err(SmeltError::AiResolution { .. })`:
     - Print the error: `eprintln!("AI resolution failed: {message}")`
     - Print "Falling back to manual resolution..."
     - Restore the original conflicted files
     - Delegate to the existing `InteractiveConflictHandler`

**4. Update `execute_merge_run`** to accept `no_ai: bool` parameter and construct the appropriate handler:
- If `no_ai` or no TTY: use `InteractiveConflictHandler` (same as before)
- Otherwise: load `AiConfig` from `.smelt/config.toml`. If `config.enabled` is false, use `InteractiveConflictHandler`. Otherwise, construct `GenAiProvider` from config, then `AiConflictHandler`, then `AiInteractiveConflictHandler`.
- If `GenAiProvider::new()` fails (no API key, network issue): log warning, fall back to `InteractiveConflictHandler`.
- Because the two handler types are different, use a trait object `Box<dyn ConflictHandler>` â€” BUT `ConflictHandler` uses RPITIT which doesn't support `dyn`. **Workaround:** Use an enum dispatcher:
  ```rust
  enum MergeConflictHandler<G: GitOps, P: AiProvider> {
      AiInteractive(AiInteractiveConflictHandler<G, P>),
      Interactive(InteractiveConflictHandler),
  }
  ```
  Implement `ConflictHandler` for the enum, dispatching to the inner handler. This avoids the RPITIT-no-dyn issue.

  Actually, since `GenAiProvider` is the only `AiProvider` impl, the types are concrete. The enum can be:
  ```rust
  enum MergeConflictHandler {
      AiInteractive(AiInteractiveConflictHandler<GitCli, GenAiProvider>),
      Interactive(InteractiveConflictHandler),
  }
  ```
