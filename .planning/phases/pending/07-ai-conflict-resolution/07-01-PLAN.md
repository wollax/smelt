**1. Create `crates/smelt-core/src/ai/provider.rs`** with `GenAiProvider`:

```rust
pub struct GenAiProvider {
    client: genai::Client,
}
```

- `GenAiProvider::new(config: &AiConfig) -> crate::Result<Self>`:
  - Build a `genai::Client`. If `config.endpoint` is set, use `genai::ClientConfig` with a custom `ServiceTargetResolver` or the builder pattern to set the base URL. Use Context7 or genai docs to determine the exact API for custom endpoints — the genai crate uses `ClientConfig::default().with_chat_options(...)` or similar builder.
  - If `config.api_key` is set, configure it via the appropriate genai mechanism (likely environment-variable passthrough or client config). Check genai docs — many providers read `ANTHROPIC_API_KEY`, `OPENAI_API_KEY` etc. from env automatically, so api_key in config may just need to be set as an env var override.
  - Map genai client construction errors to `SmeltError::AiResolution`.

- `impl AiProvider for GenAiProvider`:
  - `complete()`: Build a `ChatRequest` with `ChatMessage::system(system_prompt)` and `ChatMessage::user(user_prompt)`. Call `self.client.exec_chat(model, chat_req, None).await`. Map genai errors to `SmeltError::AiResolution { message }`. Extract first text from response. Return error if response is empty.
  - **Post-processing:** Strip markdown code fences from response if present. Use a helper `strip_code_fences(s: &str) -> String` that removes leading ```` ```lang\n ```` and trailing ```` \n``` ```` patterns. This is critical — LLMs often wrap output in fences despite instructions.

- Unit tests:
  - `strip_code_fences` handles: no fences (passthrough), fences with language tag, fences without language tag, fences with extra whitespace.

**2. Create `crates/smelt-core/src/ai/prompt.rs`** with prompt construction:

- `build_system_prompt() -> &'static str`: Returns the system prompt instructing the LLM to output only raw file content. Content:
```
You are resolving a git merge conflict. Output ONLY the complete resolved file content.

Rules:
- No markdown fences, no explanations, no commentary
- Output the COMPLETE file, not a diff or patch
- Preserve all functionality from both versions
- If changes are in different parts of the file, include both
- If changes conflict on the same lines, integrate both intents
- Maintain consistent code style with the surrounding context
```

- `build_resolution_prompt(file_path: &str, base: &str, ours: &str, theirs: &str, session_name: &str, task_desc: Option<&str>, commit_subjects: &[String]) -> String`:
  Builds the user prompt with 3-way context:
```
File: {file_path}
Session: {session_name}
Task: {task_desc or "(none)"}
Recent commits: {commit_subjects joined with newlines, or "(none)"}

## Base version (common ancestor)
{base_content}

## Current version (ours - target branch)
{ours_content}

## Incoming version (theirs - session branch)
{theirs_content}

Resolve the conflict and output the complete merged file.
```

- `build_retry_prompt(original_prompt: &str, feedback: &str) -> String`:
  Appends user feedback to the original prompt:
```
{original_prompt}

## User feedback on previous attempt
{feedback}

Please resolve again incorporating this feedback.
```

- Unit tests:
  - `build_resolution_prompt` includes file path, session name, all three content sections.
  - `build_retry_prompt` includes original prompt and feedback.
  - `build_system_prompt` contains "ONLY" and "no markdown".
