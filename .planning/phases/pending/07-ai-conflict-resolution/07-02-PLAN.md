**1. Create `crates/smelt-core/src/merge/ai_handler.rs`** with `AiConflictHandler`:

```rust
use std::path::Path;
use tracing::{info, warn};

use crate::ai::{AiConfig, AiProvider, build_system_prompt, build_resolution_prompt};
use crate::error::{Result, SmeltError};
use crate::git::GitOps;
use crate::merge::conflict::{ConflictScan, scan_conflict_markers};
use crate::merge::types::ConflictAction;
use crate::merge::ConflictHandler;
```

- `AiConflictHandler<G: GitOps, P: AiProvider>`:
  - Fields: `git: G`, `provider: P`, `config: AiConfig`, `target_branch: String`
  - Constructor: `pub fn new(git: G, provider: P, config: AiConfig, target_branch: String) -> Self`

- `impl<G: GitOps, P: AiProvider> ConflictHandler for AiConflictHandler<G, P>`:
  - `handle_conflict()` implementation:
    1. Determine the model: `self.config.model.as_deref().unwrap_or(default_model_for_provider(&self.config))` where `default_model_for_provider` returns a sensible default per provider string (e.g., "claude-sonnet-4-20250514" for "anthropic", "gpt-4o" for "openai", "llama3.1" for "ollama", and a generic fallback).
    2. Get commit subjects via `self.git.log_subjects(&format!("{}..{}", &self.target_branch, "MERGE_HEAD"))`. If this fails, use empty vec (non-fatal).
    3. For each file in `files`:
       a. Extract 3-way context: `base = self.git.show_index_stage(work_dir, 1, file).await.unwrap_or_default()` (base may not exist for new files), `ours = self.git.show_index_stage(work_dir, 2, file).await?`, `theirs = self.git.show_index_stage(work_dir, 3, file).await?`
       b. Build prompt via `build_resolution_prompt(file, &base, &ours, &theirs, session_name, None, &commit_subjects)` — `task_description` is always `None` (accepted v0.1.0 limitation; could be threaded via constructor or trait extension later)
       c. Call `self.provider.complete(model, build_system_prompt(), &prompt).await`
       d. If OK: write resolved content to `work_dir.join(file)` via `tokio::fs::write`
       e. If Err: log warning, return `Err(SmeltError::AiResolution { message })` — the merge loop will propagate this, and the CLI layer can catch it and fall back
    4. After all files resolved: return `Ok(ConflictAction::Resolved(ResolutionMethod::AiAssisted))`
    5. The existing merge loop in `merge_sessions()` will re-scan for markers. If markers remain (bad AI output), it re-prompts the handler — which is the correct behavior.

- **Important:** The handler does NOT implement retry-with-feedback itself. Retry is handled at the CLI layer (Plan 03) which wraps the AI handler and the interactive handler together. The core AI handler is a single-attempt resolver. The `AiConfig.max_retries` is read by the CLI-layer wrapper.

- **Error handling:** If any file's AI resolution fails:
  - Log which file failed and why
  - Return `Err(SmeltError::AiResolution { message: format!("failed to resolve {file}: {reason}") })`
  - The CLI layer catches this and falls back to manual

- Helper function `default_model_for_provider(config: &AiConfig) -> &'static str`:
  ```rust
  fn default_model_for_provider(config: &AiConfig) -> &'static str {
      match config.provider.as_deref() {
          Some("anthropic") => "claude-sonnet-4-20250514",
          Some("openai") => "gpt-4o",
          Some("ollama") => "llama3.1",
          Some("gemini") | Some("google") => "gemini-2.0-flash",
          _ => "claude-sonnet-4-20250514", // default to Anthropic
      }
  }
  ```
