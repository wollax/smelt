Purpose: This is the central value proposition of Smelt. After this plan, the full merge pipeline works programmatically (no CLI yet). Plan 03 wires it to the CLI.
Output: MergeRunner with run() method, merge types, rollback logic, cleanup logic, and unit tests proving clean merge, conflict rollback, and session filtering.

<task type="auto">
  <name>Task 2: Implement MergeRunner with full merge pipeline</name>
  <files>
    crates/smelt-core/src/merge/mod.rs
  </files>
  <action>
    Implement `MergeRunner<G: GitOps + Clone>` in `crates/smelt-core/src/merge/mod.rs`:

    ```rust
    pub struct MergeRunner<G: GitOps + Clone> {
        git: G,
        repo_root: PathBuf,
    }
    ```

    Implement `MergeRunner::new(git: G, repo_root: PathBuf) -> Self`.

    Implement `MergeRunner::run(&self, manifest: &Manifest, opts: MergeOpts) -> Result<MergeReport>`:

    **Phase A: Preparation**
    1. Validate `.smelt/` exists → `SmeltError::NotInitialized`
    2. Read session state files from `.smelt/worktrees/<session-name>.toml` for each session in the manifest
    3. Check for Running sessions → return `SmeltError::SessionError { session, message: "session is still running — cannot merge while sessions are active" }` if any session has `SessionStatus::Running`
    4. Filter sessions to only `SessionStatus::Completed`. Log skipped sessions (Failed, Created) with `tracing::warn!("skipping session '{}': status {:?}", name, status)`
    5. If zero completed sessions → return `SmeltError::NoCompletedSessions`
    6. Collect completed session names, their branch names (from WorktreeState.branch_name), and task descriptions

    **Phase B: Target branch setup**
    7. Determine target branch name: `opts.target_branch.unwrap_or_else(|| format!("smelt/merge/{}", manifest.manifest.name))`
    8. Check target branch doesn't exist → `SmeltError::MergeTargetExists { branch }` if it does
    9. Determine base commit:
       - If 1 completed session: use `self.git.rev_parse(&session_branch_name)` to get the session's base, then walk back to find the fork point. SIMPLER: use `manifest.manifest.base_ref` resolved via `self.git.rev_parse()`.
       - ACTUALLY: Use `self.git.rev_parse(&manifest.manifest.base_ref)` to resolve the base ref. This is the commit that all sessions branched from.
       - NOTE: If base_ref is "HEAD", this resolves to the current HEAD which may have moved. The manifest's base_ref should match what was used during session creation. This is acceptable for Phase 4.
    10. Create target branch: `self.git.branch_create(&target_branch, &base_commit)`

    **Phase C: Temp worktree**
    11. Determine temp worktree path: `<repo_root>/../<repo_name>-smelt-merge-<manifest_name>`
    12. Create temp worktree: `self.git.worktree_add(&temp_path, &target_branch, &target_branch)` — WAIT, worktree_add creates a NEW branch. We need to checkout an EXISTING branch.
        - CORRECTION: Use the pattern from research: `git worktree add <path> <existing-branch>`. But the current `worktree_add` signature is `(path, branch_name, start_point)` which creates a NEW branch with `-b`.
        - SOLUTION: Add a helper method or use a different approach. The simplest: since we already created the target branch, we can use `git worktree add <path> <target_branch>` WITHOUT `-b`. This requires either:
          a. A new `worktree_add_existing` method on GitOps, or
          b. Running git directly in MergeRunner
        - BEST APPROACH: Add `worktree_add_existing(&self, path: &Path, branch_name: &str) -> Result<()>` to GitOps trait. This runs `git worktree add <path> <branch_name>` (no `-b` flag). Add this to GitOps trait and GitCli impl as part of this task (it's a simple addition).
        - ALTERNATIVE: The MergeRunner can call `self.git.worktree_add(&temp_path, "unused", &target_branch)` — but this creates a new branch "unused", which is wrong.
        - GO WITH: Add `worktree_add_existing` to GitOps + GitCli. It's 5 lines of impl.

    **Phase D: Sequential merge loop**
    13. For each completed session (in manifest order):
        a. Log progress: `tracing::info!("[{i}/{total}] Merging session '{name}'...")`
        b. Call `self.git.merge_squash(&temp_path, &session_branch_name)`
        c. On error (conflict):
           - Call `self.git.unmerged_files(&temp_path)` to get conflicting files
           - ROLLBACK:
             i. `self.git.worktree_remove(&temp_path, true)` — remove temp worktree
             ii. `self.git.branch_delete(&target_branch, true)` — delete target branch
           - Return `SmeltError::MergeConflict { session: name, files }`
        d. On success:
           - Generate commit message: `merge(<session-name>): <task-description>` (truncate task to 72 chars for subject line)
           - Body: `Session: <session-name>\nTask: <full-task-description>\nBranch: <session-branch>`
           - Call `self.git.commit(&temp_path, &commit_message)`
           - Call `self.git.diff_numstat(&format!("HEAD~1"), "HEAD")` — WAIT, this runs in repo_root. Need to run in temp worktree context.
           - CORRECTION: diff_numstat should use run_in or be called with work_dir context. ACTUALLY, diff_numstat operates on refs which are global. But we need the diff for the commit we just made on the target branch. Use: `self.git.diff_numstat(&format!("{target_branch}~{i+1}"), &format!("{target_branch}~{i}"))` — NO, this is fragile.
           - SIMPLER: After committing, capture the commit hash via the return value of `commit()`. Then `diff_numstat("{hash}~1", "{hash}")` or `diff_numstat("{hash}^", "{hash}")`.
           - BEST: Store the pre-merge HEAD and post-merge HEAD. `let pre = self.git.rev_parse("HEAD")` (in temp worktree context... but rev_parse uses repo_root). SOLUTION: Add the commit hash return from commit() and use `self.git.diff_numstat(&format!("{commit_hash}^"), &commit_hash)`.
           - Collect `MergeSessionResult` with diff stats

    **Phase E: Cleanup**
    14. Remove temp worktree: `self.git.worktree_remove(&temp_path, true)`
    15. Prune worktree metadata: `self.git.worktree_prune()`
    16. Clean up session worktrees and branches via `WorktreeManager::remove(session_name, force=true)` for each merged session
    17. Build and return `MergeReport`

    IMPORTANT: Wrap Phase D + E in a pattern where temp worktree cleanup happens on ALL exit paths (success or error). Use a helper method or explicit cleanup in error paths. Rust doesn't have `defer`, so either:
    - Use a guard struct with Drop that removes the worktree (complex)
    - Handle cleanup explicitly in each error path (simpler, already done for rollback in Phase D step c)
    - Use a helper that returns Result and cleanup after the call

    IMPORTANT: The `worktree_add_existing` method must be added to GitOps trait and GitCli. Add it right after the existing `worktree_add` method:
    ```rust
    // In GitOps trait:
    fn worktree_add_existing(&self, path: &Path, branch_name: &str) -> impl Future<Output = Result<()>> + Send;

    // In GitCli:
    async fn worktree_add_existing(&self, path: &Path, branch_name: &str) -> Result<()> {
        let path_str = path.to_string_lossy();
        self.run(&["worktree", "add", &path_str, branch_name]).await?;
        Ok(())
    }
    ```

    This helper should:

    3. Update state file to SessionStatus::Completed
    4. Return the WorktreeState
