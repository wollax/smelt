    3. Create `crates/smelt-core/src/session/manifest.rs` with these types:

       ```rust
       #[derive(Debug, Clone, Serialize, Deserialize)]
       pub struct Manifest {
           pub manifest: ManifestMeta,
           #[serde(rename = "session")]
           pub sessions: Vec<SessionDef>,
       }

       #[derive(Debug, Clone, Serialize, Deserialize)]
       pub struct ManifestMeta {
           pub name: String,
           #[serde(default = "default_base_ref")]
           pub base_ref: String,
       }

       fn default_base_ref() -> String { "HEAD".to_string() }

       #[derive(Debug, Clone, Serialize, Deserialize)]
       pub struct SessionDef {
           pub name: String,
           /// Inline task description.
           pub task: Option<String>,
           /// Path to a file containing the task description.
           pub task_file: Option<String>,
           /// Glob patterns for file scope.
           pub file_scope: Option<Vec<String>>,
           /// Override base_ref from manifest-level default.
           pub base_ref: Option<String>,
           /// Session timeout in seconds.
           pub timeout_secs: Option<u64>,
           /// Environment variables for the session.
           pub env: Option<std::collections::HashMap<String, String>>,
           /// Script definition (required for scripted sessions).
           pub script: Option<ScriptDef>,
       }

       #[derive(Debug, Clone, Serialize, Deserialize)]
       pub struct ScriptDef {
           #[serde(default = "default_backend")]
           pub backend: String,
           /// Stop after N steps (for testing partial execution).
           pub exit_after: Option<usize>,
           /// Simulate a failure mode.
           pub simulate_failure: Option<FailureMode>,
           /// Ordered list of script steps.
           pub steps: Vec<ScriptStep>,
       }

       fn default_backend() -> String { "scripted".to_string() }

       #[derive(Debug, Clone, Serialize, Deserialize)]
       #[serde(rename_all = "lowercase")]
       pub enum FailureMode {
           Crash,
           Hang,
           Partial,
       }

       #[derive(Debug, Clone, Serialize, Deserialize)]
       #[serde(tag = "action", rename_all = "lowercase")]
       pub enum ScriptStep {
           Commit {
               message: String,
               files: Vec<FileChange>,
           },
       }

       #[derive(Debug, Clone, Serialize, Deserialize)]
       pub struct FileChange {
           pub path: String,
           /// Inline file content.
           pub content: Option<String>,
           /// Path to a file with the content.
           pub content_file: Option<String>,
       }
       ```

       Implement `Manifest::load(path: &Path) -> Result<Self>`:
       - Read file, parse with `toml::from_str`, map errors to `SmeltError::ManifestParse`
       - Call `self.validate()` after parsing

       Implement `Manifest::from_str(s: &str) -> Result<Self>`:
       - Parse with `toml::from_str`, call `validate()`

       Implement `Manifest::validate(&self) -> Result<()>`:
       - Error if `sessions` is empty ("manifest must define at least one session")
       - Error if duplicate session names exist ("duplicate session name: '{name}'")
       - Error if any session has neither `task` nor `task_file` ("session '{name}' must have either 'task' or 'task_file'")
       - Error if any session has both `task` and `task_file` ("session '{name}' cannot have both 'task' and 'task_file'")
       - For each session with a script, validate steps are non-empty
       - Validate file_scope globs parse with `globset::Glob::new()` (warn-level tracing if invalid, but don't fail — allow patterns like exact paths)

       All validation errors use `SmeltError::ManifestParse`.

       Unit tests:

       - Validate rejects empty sessions
       - Validate rejects duplicate session names
       - Validate rejects session with no task or task_file
       - Round-trip: serialize then deserialize a Manifest
       - Parse FailureMode variants

  <name>Task 2: Extend GitOps with add, commit, and rev_list methods</name>

    1. Add three new methods to the `GitOps` trait in `git/mod.rs`:

       ```rust
       /// Stage files for commit. If `paths` is empty, stages all changes (`git add -A`).
       fn add(&self, work_dir: &Path, paths: &[&str]) -> impl Future<Output = Result<()>> + Send;

       /// Create a commit in the given working directory with the provided message.
       fn commit(&self, work_dir: &Path, message: &str) -> impl Future<Output = Result<String>> + Send;

       /// Count commits on `branch` that are not on `base`. Returns the count.
       fn rev_list_count(&self, branch: &str, base: &str) -> impl Future<Output = Result<usize>> + Send;
       ```

       These methods take `work_dir: &Path` because scripted sessions operate in worktrees, not the main repo root. This is different from existing methods that use `self.repo_root`.

    2. Implement in `GitCli` (`git/cli.rs`):

       For `add`: Run `git -C <work_dir> add <paths...>` or `git -C <work_dir> add -A` if paths is empty. Use `Command::new(&self.git_binary)` with `.current_dir(work_dir)` — do NOT use `self.run()` since that uses `self.repo_root`.

       For `commit`: Run `git -C <work_dir> commit -m <message>`. Return the short commit hash by parsing stdout or running a follow-up `git -C <work_dir> rev-parse --short HEAD`.

       For `rev_list_count`: Run `git rev-list --count <branch> --not <base>`. Use `self.repo_root` as cwd since this operates on refs visible from the main repo.

       IMPORTANT: For `add` and `commit`, do NOT use the existing `run()` helper because it hardcodes `current_dir(&self.repo_root)`. Either create a `run_in(work_dir, args)` helper or inline the Command construction. The `run_in` approach is recommended to reduce duplication:

       ```rust
       async fn run_in(&self, work_dir: &Path, args: &[&str]) -> Result<String> {
           // Same as run() but uses work_dir instead of self.repo_root
       }
       ```

    3. Add unit tests in `cli.rs` (extend the existing `setup_test_repo` helper):
       - Test `add` + `commit`: create a file in the repo, stage it, commit it, verify rev_list_count shows 1 more commit
       - Test `commit` returns a valid short hash
       - Test `rev_list_count` returns 0 for branches at the same point, and N for branches with N additional commits
       - Test `add` with specific file paths (not empty/all)
       - Test `add` and `commit` work in a worktree directory (create a worktree, write a file, add, commit in the worktree path)
