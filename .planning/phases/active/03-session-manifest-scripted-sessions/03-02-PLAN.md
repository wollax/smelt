  <files>

  </files>
  <action>
    Create `crates/smelt-core/src/session/script.rs` with:

    ```rust
    pub struct ScriptExecutor<'a, G: GitOps> {
        git: &'a G,
        worktree_path: PathBuf,
    }
    ```

    Implement `ScriptExecutor::new(git: &'a G, worktree_path: PathBuf) -> Self`.

    Implement `ScriptExecutor::execute(&self, script: &ScriptDef) -> Result<SessionResult>`:

    1. Record `start_time = Instant::now()`
    2. Determine effective step count: `let max_steps = script.exit_after.unwrap_or(script.steps.len()).min(script.steps.len())`
    3. Iterate over `script.steps[..max_steps]`:
       - For `ScriptStep::Commit { message, files }`:
         a. Write each file to the worktree:
            - If `content` is Some, write it with `std::fs::write(worktree_path.join(&file.path), content)`
            - If `content_file` is Some, read the content from that path and write it
            - If neither, create an empty file
            - Create parent directories with `std::fs::create_dir_all` as needed
         b. Collect file paths into a `Vec<&str>`
         c. Call `self.git.add(&self.worktree_path, &file_paths).await?`
         d. Call `self.git.commit(&self.worktree_path, message).await?`
         e. Increment `steps_completed`
       - Handle failure simulation:
         - `FailureMode::Crash`: after completing `max_steps`, return `SessionOutcome::Failed` with reason "simulated crash"
         - `FailureMode::Hang`: NOT implemented in v0.1.0 — return `SessionOutcome::Failed` with reason "hang simulation not implemented" (it would require sleeping indefinitely, which isn't useful for in-process execution)
         - `FailureMode::Partial`: execute the Commit step but only write/add HALF the files (first N/2), then return `SessionOutcome::Failed` with reason "simulated partial failure"
    4. If no failure simulation, return `SessionOutcome::Completed`
    5. Populate `SessionResult` with: steps_completed, has_commits (steps_completed > 0), duration, outcome, failure_reason

    IMPORTANT: File I/O (std::fs::write, create_dir_all) MUST be synchronous — do NOT use tokio::fs. These are small files and tokio::fs adds unnecessary complexity. The git operations (add, commit) are async.

    IMPORTANT: When handling `FailureMode::Partial` in a Commit step, you need to decide which files to include. Use `files[..files.len()/2].max(1)` to ensure at least one file is written. The commit should still succeed (with fewer files), but the step is marked as failed.

    Update `session/mod.rs` to add `pub mod script;` and re-export `ScriptExecutor`.

    Unit tests (using the real `GitCli` + temp repo, following the pattern from `worktree/mod.rs` tests):
    - Test executing a 2-step script creates 2 commits (use rev_list_count to verify)
    - Test `exit_after = 1` on a 3-step script creates only 1 commit
    - Test `simulate_failure = Crash` with `exit_after = 1` creates 1 commit and returns Failed
    - Test files are actually written to the worktree path
    - Test parent directories are created for nested file paths (e.g., `src/auth/login.rs`)
  </action>

  </verify>
  <done>
    ScriptExecutor creates real git commits in worktrees. exit_after truncates execution. Failure modes return appropriate outcomes. Files are written with parent directory creation.
  </done>

<task type="auto">
  <name>Task 2: Implement SessionRunner</name>
  <files>
    crates/smelt-core/src/session/runner.rs
    crates/smelt-core/src/session/mod.rs
  </files>
  <action>
    Create `crates/smelt-core/src/session/runner.rs` with:

    ```rust
    pub struct SessionRunner<G: GitOps> {
        git: G,
        repo_root: PathBuf,
    }
    ```

    Implement `SessionRunner::new(git: G, repo_root: PathBuf) -> Self`.

    Implement `SessionRunner::run_manifest(&self, manifest: &Manifest) -> Result<Vec<SessionResult>>`:

    1. Validate that `.smelt/` exists (return `SmeltError::NotInitialized` if not)
    2. Create a `WorktreeManager` from `self.git` and `self.repo_root`
       - NOTE: WorktreeManager takes ownership of git. Since SessionRunner also needs git, the runner should take `&G` where G: GitOps + Clone, OR the runner should create worktrees first, then run scripts. The simplest approach: SessionRunner holds the git instance and creates a fresh WorktreeManager by reference.
       - ACTUALLY: WorktreeManager<G> takes G by value. The cleanest approach is for SessionRunner to NOT own a WorktreeManager, but instead receive a `&WorktreeManager<G>` or take `G: GitOps + Clone`. Given the existing code, use `G: GitOps + Clone` — `GitCli` can easily derive Clone (it only holds PathBuf fields).
       - FIRST: Add `#[derive(Clone)]` to `GitCli` in `cli.rs`.
    3. For each session in `manifest.sessions`:
       a. Determine `base_ref`: session-level `base_ref` overrides manifest-level
       b. Create a worktree via `WorktreeManager::create()` with:
          - `session_name`: session.name
          - `base`: effective base_ref
          - `task_description`: session.task.clone()
          - `file_scope`: session.file_scope.clone()
       c. If the session has a `script`:
          - Create a `ScriptExecutor` with the git instance and worktree path
          - Call `executor.execute(&script)` to get the result
          - Collect the SessionResult
       d. If no script, return a SessionResult with outcome Completed, 0 steps, has_commits false
    4. Return all results as a Vec

    IMPORTANT: Sessions run SEQUENTIALLY in this plan. Parallel execution (tokio::spawn) is deferred — it adds complexity around git locking and error propagation. Sequential is correct and sufficient for scripted sessions.

    IMPORTANT: The SessionRunner does NOT clean up worktrees on failure. Worktrees persist for inspection. The user can run `smelt worktree prune` to clean up.

    Update `session/mod.rs` to add `pub mod runner;` and re-export `SessionRunner`.

    Unit tests:
    - Test run_manifest with a 2-session manifest: both sessions create commits in separate worktrees
    - Test run_manifest returns correct SessionResults (one per session)
    - Test that two sessions editing the same file path create separate commits in separate worktrees (verify both branches exist, both have commits)
    - Test run_manifest with NotInitialized error when .smelt/ doesn't exist
    - Test conflict setup: two sessions both write to `src/lib.rs` with different content — verify both branches have the file with different content (this sets up the merge conflict scenario for Phase 4)
  </action>

  </verify>
  <done>
    SessionRunner coordinates worktree creation + script execution for all sessions in a manifest. Two sessions writing the same file produce branches that will conflict on merge. All results collected and returned. Sequential execution works correctly.
  </done>
